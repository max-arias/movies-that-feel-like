## Destination

An end-to-end run of the pipeline on **all reachable posts from `r/MoviesThatFeelLike`** (fetch → normalize → cache-assets → extract → enrich → load) completes in well under 1 hour AND outputs a versioned data migration file that the production Cloudflare D1 can apply via `wrangler d1 migrations apply --remote`. **The original "1k-post" target was reduced to "all reachable posts"** after empirical discovery during ticket 17: Arctic Shift caps each query at ~100 posts (no pagination exposed by `arcshiftwrap`) and the subreddit has ~240-340 reachable posts across 2020, 2024, 2025, and 2026 (other years are empty). The pipeline runs only locally — prod never runs the pipeline itself. The data migration lives in `packages/db/migrations/`, sequenced after the schema migrations (`0001_initial.sql` … `0004_drop_tmdb_data.sql`), is idempotent on re-apply (uses `INSERT OR IGNORE` semantics so the same file can be re-applied safely), and is auto-committed and pushed to git by the pipeline. Verified by an actual end-to-end run on real Reddit data from `r/MoviesThatFeelLike` (2020 + 2024 + 2025 + 2026), and by a clean `wrangler d1 migrations apply --remote` of the emitted file against a fresh prod-shaped D1.

## Notes

Domain: the pipeline currently lives in `apps/pipeline/src/pipeline/` and is invoked through root-level npm scripts. Each stage is a separate CLI. The pipeline runs locally; the loader writes to `data/app.db` and emits a versioned data migration to `packages/db/migrations/` (sequenced after the schema migrations, default filename `NNNN_seed_<UTC-timestamp>.sql`). The data migration is the **only** SQL output of the load stage — no `--sql-out` flag, no chunked directory, no separate `d1:apply` step. `wrangler d1 migrations apply` (with `--local` for local D1, `--remote` for prod D1) is the single apply path for both environments. Wrangler's built-in migration tracking (it remembers which migrations have been applied to a given DB) handles the "this migration was already applied, skip" case. The data migration is idempotent because every `INSERT` is `INSERT OR IGNORE`, keyed by the table's natural unique constraint (`reddit_post_id` on `imported_vibe_posts`, the `(recommendation_id, imported_vibe_post_id, evidence_comment_id)` triple on `recommendation_evidence`, the `(imported_vibe_post_id, tag)` pair on `vibe_tags`).

Skills every session should consult:

- `code-review` — before closing refactor tickets, since the changes touch several stages.
- `diagnosing-bugs` — when a 1k-stage run fails or a refactor regresses correctness.
- `wrangler` — central to the new direction: `wrangler d1 migrations apply <DB> --local` for local D1, `--remote` for prod D1. The deploy step will use this.
- `cloudflare` — only if a ticket starts considering D1/R2 architecture beyond what wrangler already provides.
- `verification-planning` — for the 1k verification ticket, since the acceptance signal is now "under 1h AND clean remote apply" rather than just wall-clock.

Standing preferences from the destination-grilling session:

- **Comprehensive async throughout** — `asyncio` + `httpx.AsyncClient` for `fetch.py`, `cache_assets.py`, `enrich.py`. `extract.py` keeps its existing thread-pool + checkpointed design (Instructor's `client.create()` is sync); the change there is just the rate-limit + concurrency defaults, not a rewrite.
- **LLM rate limit 60 RPM**, concurrency 8 in `extract.py`. Landed in ticket 11.
- **Provider cache on disk** — JSON cache file per provider (TMDB, IGDB) keyed by `candidate_key`. Landed in ticket 12.
- **Data migrations co-located in `packages/db/migrations/`** — sequenced after the schema migrations (`0001_initial.sql` … `0004_drop_tmdb_data.sql`) so `wrangler d1 migrations apply` picks them up in the right order. Migration file naming: `NNNN_seed_<UTC-timestamp>.sql` (e.g. `0005_seed_20260714T1500Z.sql`).
- **Pipeline auto-commits and pushes** the new migration file. The commit message is generated (e.g. `data: 1k-post run, 0005_seed_20260714T1500Z.sql`). No developer review gate.
- **Idempotent data SQL** — every `INSERT` is `INSERT OR IGNORE`. The unique constraints in the schema (`reddit_post_id`, `(recommendation_id, imported_vibe_post_id, evidence_comment_id)`, `(imported_vibe_post_id, tag)`) are the natural keys. `recommendations` has no unique constraint, so the loader's existing tmdb-id / igdb-id + media_type lookup is preserved (the migration SQL uses `INSERT OR IGNORE` plus a unique index on `(tmdb_id, media_type)` for movie/TV and `(igdb_id, media_type)` for game — these indexes already exist per `0001_initial.sql` and `0003_games.sql`).
- **Keep image downloads in the 1k run** — they gate publishability in `_has_usable_image`. The UI hotlinks `source_url`; the local `cache_key` (currently a local path) is written into the migration as-is and ignored by the UI. No R2 migration work in this map.
- **Multi-year data for the end-to-end run** — current 2026 archive is ~100 posts; backfill earlier years (2025, 2024, 2020) to maximize coverage. Total reachable is ~240-340 posts. Fetched as separate yearly artifacts, merged in `normalize` (ticket 17).
- **No standalone profile artifact** — measurements live in commit messages, ticket resolutions, and the 1k run's wall-clock output.
- **Verification** — actually run all reachable posts end-to-end after the refactors land; the run's wall-clock is the runtime acceptance signal AND the emitted migration's `wrangler d1 migrations apply --remote` against a fresh prod-shaped D1 is the deploy acceptance signal. Both must pass. The "under 1 hour" budget is now trivially achievable at this scale (300 posts vs the original 1k target).

The destination's mention of `httpx.AsyncClient` for fetch.py is therefore superseded by the to_thread plan (enrich.py in ticket 15 will be the one that genuinely needs httpx). **(Knobs later dropped — see the "Operational vs hardcoded knobs" note below the index.)**

Want a per-stage `T0` / `T1` print so we can tell which stage ate the budget if 1k comes in at 55 min instead of 30 min. Probably a 5-line change in each stage's `main()`. - **TMDB/IGDB rate-limit/backoff policy at higher concurrency** — when 1k posts × 3 unique candidates = 3000 lookups fire at 8-16 parallel, TMDB and IGDB may 429. The existing `enrich._run_with_retry` is per-call; we need a global token-bucket limiter similar to `extract._RequestLimiter`. Will surface during *Refactor enrich.py to async + integrate provider cache*. Fog for now.
- **Migration file size and chunking at 10k+** — at 1k posts the data migration is probably a few MB; `wrangler d1 migrations apply` should handle that fine. At 10k+ it might hit a practical limit, in which case chunking (one migration per chunk, sequenced) would be needed. Fog — not in this map's route unless 1k verification surfaces it.
- **CI / CD workflow for the auto-push** — the pipeline auto-commits and pushes the migration. That requires a git identity in the local environment (already present), a clean working tree, and an `origin` remote with write access. Worth confirming as part of the auto-push ticket. Not a separate ticket; lives in the migration-emit ticket.
- **Memory pressure at 1k** — the 100-post test fit comfortably in memory. The 1k load step will read all four artifacts (normalized, assets, extraction, enrichment) into memory in `_build_merge_index`; with 1k posts × ~3 candidates × evidence lists, the extraction artifact could be 50-100 MB. SQLite load is fine. Worth measuring during verification but probably not a real problem.
- **Unique indexes on `recommendations(tmdb_id, media_type)` / `(igdb_id, media_type)`** — surfaced in ticket 19's resolution. The destination text says these are "already unique indexes per 0001_initial.sql and 0003_games.sql" but the schema only has plain `idx_recommendations_tmdb_id` / `idx_recommendations_igdb_id` (not UNIQUE). `INSERT OR IGNORE` in the emitted migration is therefore a no-op for `recommendations` re-apply (the only real-world re-apply is via wrangler's tracking, which already prevents re-apply). If a manual re-apply workflow ever materializes, a new schema migration (`0006_add_unique_rec_indexes.sql`) before the seed migrations would harden it. Not a blocker for ticket 19; lives in the fog until/unless someone needs manual re-apply.

## Out of scope

- A re-architecture of the pipeline into a long-running service (e.g. a Cloudflare Worker that ingests incrementally). The destination is a one-shot 1k run; a streaming service is a different shape of work.
- Switching the LLM provider. The model is `deepseek-v4-flash` on OpenCode Go; cheaper-than-LLM is sufficient at 60 RPM.
- Switching the asset source. Reddit CDN URLs are what they are; the cache-failure rate (~15% on the 100-post sample) is a known pre-existing issue surfaced in the handoff doc and is not in this map's route.
- UI changes. The site renders whatever D1 has; the 1k run produces more rows, not different ones.
- R2 / image hosting. The UI hotlinks `source_url`; the `cache_key` column carries a local path that's harmless in prod. No R2 migration work in this map. (Future effort if/when the prod site wants to self-host images.)
- Auto-deploy from the pipeline push. The pipeline pushes to git; `npm run cf:deploy` (run separately or in CI) is what applies the migration to remote D1 and deploys the worker. The pipeline itself never calls `wrangler deploy --remote`.
- A standalone profile / benchmark writeup. The 1k verification is the profile, written into the verification ticket's resolution.
